# 老师主动接单功能规格

## 概述

开放需求大厅供老师主动浏览并申请家长发布的需求，形成「系统自动匹配推送 + 老师主动申请」的双轨模式，增加老师的参与感和订单机会，同时让家长能从更多候选老师中做出更好的选择，提升平台整体成单率。

---

## 现有架构

当前平台仅有单向的系统匹配推送流程：家长发布需求 → 匹配算法（技能 25pt + 位置 30pt + 价格 25pt + 时间 20pt）计算分数 → 向匹配老师发送通知（`notifications` 表，`type = 'match_demand'`）→ 匹配结果记录在 `match_logs` 表。

老师只能被动接收通知，无法主动浏览需求列表，也无法针对特定需求主动申请，错失对自己有特别优势的需求。

---

## 功能详情

### 用户流程

#### 老师端：浏览并申请需求

1. 老师进入「需求大厅」（`/teacher/demand-hall`）
2. 浏览所有 `status = 'open'` 的需求列表（家长信息脱敏显示）
3. 可按**服务类别**、**城市**、**预算范围**筛选
4. 点击某需求查看详情（含服务类型、时间偏好、预算范围、孩子年龄、特殊要求）
5. 点击「申请接单」，填写：
   - 自我介绍（针对该需求的匹配理由，≤ 500 字）
   - 报价（`/小时`，需在家长预算范围内，或可超出但需说明）
6. 提交申请，进入「待家长审核」状态
7. 收到家长接受/拒绝通知
8. 被接受后，系统自动创建订单（或引导双方沟通确定订单细节）

**已申请限制**：
- 同一老师对同一需求只能申请一次
- 老师每日最多申请 10 个需求（防刷）
- 已被其他老师成功接单（`demand.status != 'open'`）的需求不可申请

#### 家长端：查看并处理申请

1. 家长在「我的需求」中，若某需求有新申请，显示「N 个老师申请」角标
2. 点击进入「申请列表」，展示所有申请该需求的老师卡片：
   - 老师姓名（全名）、头像、认证状态
   - 老师评分、历史订单数
   - 申请报价
   - 自我介绍摘要
3. 点击老师卡片查看完整申请详情
4. 操作：
   - **接受**：该申请变为 `accepted`，其余申请自动变为 `rejected`（系统通知），需求状态更新为 `matched`
   - **拒绝**：该申请变为 `rejected`，通知老师
5. 接受后，引导家长与老师进入聊天或直接创建订单

### 数据库设计

#### 新增表：`demand_applications`

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | INTEGER PK | 自增主键 |
| `demand_id` | INTEGER NOT NULL | references `demands.id` |
| `teacher_id` | INTEGER NOT NULL | references `users.id`（申请老师） |

- `CREATE INDEX idx_demand_applications_demand ON demand_applications(demand_id, status);`
- `CREATE INDEX idx_demand_applications_teacher ON demand_applications(teacher_id, created_at DESC);`

## 技术方案

### 与现有匹配推送的关系

两套机制互补，不冲突：

| 维度 | 系统自动推送 | 老师主动申请 |
|---|---|---|
| **触发时机** | 家长发布需求后即时触发 | 老师随时浏览大厅后主动触发 |
| **筛选标准** | 匹配算法（评分 Top N） | 老师自主判断是否适合 |
| **家长体验** | 被动等待推荐老师联系 | 主动浏览候选老师申请 |
| **老师体验** | 被动收到通知 | 主动发现并申请感兴趣的需求 |
| **数据表** | `match_logs` + `notifications` | `demand_applications` |
| **互动**  | 老师看到通知后可进入详情页 | 老师提交申请，家长审核 |

**协同机制**：系统推送的老师和主动申请的老师都会出现在家长的申请列表中，由家长统一审核。

### 申请状态流转

```
老师提交申请
    ↓
pending（待家长审核）
    ├→ accepted（家长接受）→ 触发 demand.status = 'matched'，其余申请自动 rejected
    ├→ rejected（家长拒绝）→ 通知老师
    └→ withdrawn（老师撤回，仅 pending 状态可操作）
```

### 每日限额实现

```typescript
// 使用当日零点到现在的申请数做判断
const today = new Date();
today.setHours(0, 0, 0, 0);

const todayCount = await db.select({ count: sql`count(*)` })
  .from(demandApplications)
  .where(and(
    eq(demandApplications.teacherId, teacherId),
    gte(demandApplications.createdAt, today)
  ));

if (todayCount[0].count >= 10) {
  throw new Error('每日申请上限为 10 个');
}
```

## 安全考虑

- **家长信息保护**：需求大厅 API 绝不返回 `parentId`、家长姓名、手机号等身份信息，只在家长接受申请后，才通过聊天系统开放沟通渠道
- **权限验证**：只有 `role = 'teacher'` 的用户才能访问需求大厅和提交申请；只有需求的 `parentId` 对应用户才能查看申请列表和操作
- **防刷申请**：每日限额 10 个 + 同需求唯一约束，防止老师批量刷申请骚扰家长
- **幂等性**：接受申请的 API 需在事务中执行（更新申请状态 + 更新需求状态 + 拒绝其他申请），防止并发接受多个申请

## Qclaw 集成方案（Phase 2）

### 背景

后期计划让老师端的 Qclaw（基于 OpenClaw 的微信端 AI Agent）自动从外部平台抓取兼职需求，回传到智慧桥平台的任务池，实现「平台内需求 + 外部抓取需求」双来源供给。

### 架构设计

```
外部信息源（58同城、本地群、豆瓣等）
        ↓  Qclaw Skill 抓取 + 解析
        ↓
   Qclaw Agent（跑在老师/运营电脑上）
        ↓  调用平台 API 回传结构化数据
        ↓
   智慧桥后端（统一任务池）
        ↓  匹配算法 + 分发
        ↓
   老师端通知 / 需求大厅展示
```

### 为 Qclaw 预留的 API

| Method | Path | 描述 |
|---|---|---|
| `POST` | `/api/external/demands` | Qclaw 提交外部抓取的需求（需 API Key 认证） |
| `GET` | `/api/external/demands/status` | Qclaw 查询已提交需求的状态 |
| `POST` | `/api/external/demands/:id/apply` | Qclaw 代替老师自动申请（需绑定老师账号） |
| `GET` | `/api/external/task-pool` | Qclaw 拉取待处理的任务列表 |
| `POST` | `/api/external/heartbeat` | Qclaw 节点心跳上报（在线状态 + 版本） |

### Qclaw Skill 类型规划

| Skill 名称 | 功能 | 阶段 |
|---|---|---|
| `demand-crawler-58` | 从58同城抓取家教/陪伴类兼职信息 | Phase 2a |
| `demand-crawler-douban` | 从豆瓣本地组抓取相关需求 | Phase 2a |
| `demand-parser` | 用 LLM 将非结构化文本解析为标准字段 | Phase 2a |
| `auto-matcher` | 根据老师偏好自动筛选并申请匹配需求 | Phase 2b |
| `auto-reply` | 生成定制化申请话术并代老师提交 | Phase 2b |

### 安全与合规

- **API Key 认证**：每个 Qclaw 节点分配唯一 API Key，所有外部 API 需在 Header 中携带 `X-Qclaw-Key`
- **抓取频率限制**：每个节点每小时最多提交 50 条外部需求，防止被源平台风控
- **内容审核**：外部需求入库后默认 `pending` 状态，由运营审核通过后才进入需求大厅
- **去重机制**：通过 `source + source_id` 联合唯一约束 + 标题相似度检测双重去重
- **隐私保护**：外部需求中的联系方式仅运营后台可见，不暴露给老师端
