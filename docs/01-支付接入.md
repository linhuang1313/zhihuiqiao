# 真实支付接入功能规格

## 概述

将现有的人工确认收款模式替换为微信支付与支付宝的自动化支付流程，覆盖**解锁套餐购买**和**订单服务费支付**两大场景，实现资金实时到账、平台自动确权，消除管理员手动对账的运营瓶颈，提升家长和老师的交易体验。

---

## 现有架构

当前支付流程依赖 `user_purchases` 表中的 `status` 字段（`pending/confirmed/expired/refunded`）以及 `confirmedBy` 和 `confirmedAt` 字段，由管理员在后台手动将 `pending` 订单标记为 `confirmed`。`orders` 表的 `paymentStatus` 同样为手动更新。该模式存在延迟高、容易出错、无法支持高并发的缺陷。

---

## 功能详情

### 用户流程

#### 场景 A：购买解锁套餐

1. 家长在套餐购买页选择套餐
2. 点击「立即购买」，系统在后端创建 `payments` 支付记录（状态 `created`）及 `user_purchases` 记录（状态 `pending`）
3. 前端展示支付方式选择弹窗（微信支付 / 支付宝）
4. 选择后，后端调用对应支付 API 生成支付二维码或跳转链接
5. 前端展示二维码（PC）或调起支付收银台（H5/小程序）
6. 前端每 2 秒轮询支付状态接口，最长 10 分钟
7. 支付平台异步回调后端 Webhook → 验签 → 更新 `payments.status = 'paid'` → 更新 `user_purchases.status = 'confirmed'` → 发送通知
8. 前端轮询到已支付 → 跳转成功页，显示剩余解锁次数

#### 场景 B：订单服务费支付

1. 家长确认订单信息后点击「去支付」
2. 系统创建 `payments` 记录，关联 `orders.id`
3. 后续流程与场景 A 步骤 3–8 相同
4. 支付成功后 `orders.paymentStatus = 'paid'`，触发老师收到「订单已付款」通知

#### 场景 C：退款

1. 家长/管理员在后台发起退款申请
2. 系统调用支付平台退款 API
3. 异步回调更新 `payments.status = 'refunded'`，同步更新 `user_purchases.status = 'refunded'` 或 `orders.paymentStatus = 'refunded'`
4. 向家长发送退款到账通知

### 数据库设计

#### 新增表：`payments`

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | INTEGER PK | 自增主键 |
| `out_trade_no` | TEXT UNIQUE NOT NULL | 平台内部支付流水号（UUID） |
| `third_party_trade_no` | TEXT | 第三方交易流水号（微信/支付宝返回） |
| `channel` | TEXT NOT NULL | 支付渠道：`wechat` / `alipay` |
| `scene` | TEXT NOT NULL | 支付场景：`unlock_package` / `order` |
| `related_id` | INTEGER NOT NULL | 关联 ID（`user_purchases.id` 或 `orders.id`） |
| `user_id` | INTEGER NOT NULL | 付款用户 ID，references `users.id` |
| `amount` | REAL NOT NULL | 支付金额（元） |
| `status` | TEXT NOT NULL DEFAULT `'created'` | `created` / `paid` / `failed` / `refunded` / `closed` |
| `raw_callback` | TEXT | 支付平台原始回调 JSON（用于审计） |
| `paid_at` | INTEGER | 支付成功时间戳 |
| `refund_amount` | REAL | 退款金额 |
| `refunded_at` | INTEGER | 退款时间戳 |
| `created_at` | INTEGER | 创建时间戳 |
| `expired_at` | INTEGER | 支付过期时间（通常 15 分钟） |

#### `user_purchases` 表变更

| 变更类型 | 字段 | 说明 |
|---|---|---|
| 新增 | `payment_id` | references `payments.id`，关联支付记录 |
| 移除依赖 | `confirmedBy` | 不再需要人工确认字段（保留历史数据兼容，不删除） |

#### `orders` 表变更

| 变更类型 | 字段 | 说明 |
|---|---|---|
| 新增 | `payment_id` | references `payments.id`，关联支付记录 |

### API 设计

| Method | Path | 描述 |
|---|---|---|
| `POST` | `/api/payments/create` | 创建支付订单，返回二维码 URL 或支付链接 |
| `GET` | `/api/payments/:outTradeNo/status` | 查询支付状态（前端轮询用） |
| `POST` | `/api/payments/wechat/callback` | 微信支付异步回调（Webhook，无需 auth） |
| `POST` | `/api/payments/alipay/callback` | 支付宝异步回调（Webhook，无需 auth） |
| `POST` | `/api/payments/:outTradeNo/refund` | 发起退款（需管理员或本人权限） |
| `GET` | `/api/payments/history` | 查询当前用户支付历史 |
| `GET` | `/api/admin/payments` | 管理员查询所有支付流水（支持筛选） |

**创建支付订单请求体：**

```typescript
{
  channel: 'wechat' | 'alipay',
  scene: 'unlock_package' | 'order',
  relatedId: number,     // packageId 或 orderId
  clientType: 'web' | 'h5' | 'miniprogram'
}
```

**创建支付订单响应体：**

```typescript
{
  outTradeNo: string,
  qrCodeUrl?: string,       // PC 扫码场景
  payUrl?: string,           // H5 跳转场景
  miniProgramParams?: object // 小程序调起支付参数
}
```

## 技术方案

### 方案对比

| 维度 | 自建微信/支付宝直连 | 聚合支付（Ping++ / 收钱吧 / 乐刷） |
|---|---|---|
| **接入成本** | 高：两套 SDK，两套签名，两套证书 | 低：统一 SDK，一次对接 |
| **维护成本** | 高：独立跟进微信/支付宝 API 变更 | 低：聚合平台统一维护 |
| **费率** | 微信 0.6%，支付宝 0.6% | 聚合平台额外加收 0.1%–0.3% |
| **到账时效** | T+1 | T+1（透传） |
| **合规要求** | 需直接申请微信/支付宝商户号 | 需在聚合平台开户，同样需营业执照 |
| **功能支持** | 原生全功能 | 覆盖主流功能，边缘需求可能受限 |
| **上线周期** | 2–3 周（含审核） | 1–2 周 |
| **适用阶段** | 成熟期、交易规模大 | 早期/中期，快速验证 |

**推荐方案：聚合支付（Ping++）**

理由：
- 平台处于早期阶段，优先速度和工程效率
- Ping++ 文档完善，Node.js SDK 成熟
- 一套代码同时支持微信 Native/JSAPI/H5 和支付宝 PC/H5
- 后期可无缝切换直连，数据库设计已留 `channel` 字段

**Ping++ 集成要点：**

```typescript
// server/services/payment.ts
import Pingpp from 'pingpp-node';
Pingpp.setApiKey(process.env.PINGPP_API_KEY);

export async function createCharge(params: CreateChargeParams) {
  const charge = await Pingpp.charges.create({
    order_no: params.outTradeNo,
    amount: Math.round(params.amount * 100), // 分
    channel: params.channel, // 'wx_native' | 'alipay_pc_direct'
    currency: 'cny',
    subject: params.subject,
    body: params.body,
    client_ip: params.clientIp,
    app: { id: process.env.PINGPP_APP_ID },
  });
  return charge;
}
```

## 安全考虑

### 签名验证
- 所有支付平台回调必须验签，拒绝无效签名请求
- Ping++ 使用 RSA 公钥验证 Webhook 签名：`X-Pingplusplus-Signature` Header
- 微信直连使用 HMAC-SHA256 验证
- 验签逻辑在独立中间件中实现，与业务逻辑解耦

### 幂等性
- 以 `out_trade_no` 为幂等键，回调处理前先查库
- 使用数据库事务 + `status` 字段 CAS（Compare-And-Swap）操作：仅 `created` 状态可更新为 `paid`
- 回调接口必须在 5 秒内返回 `200`，业务逻辑异步处理

### 防重复支付
- 创建支付订单时检查：同一 `relatedId` + `scene` 是否已存在 `created` 或 `paid` 状态的记录
- 如存在 `created` 且未过期，直接返回原有支付链接，不新建
- 前端支付按钮防抖 + loading 状态锁

### 金额校验
- 后端创建 Charge 时从数据库实时查询金额，**不信任**前端传入金额
- 回调中对比 `payments.amount × 100` 与第三方回调金额，不一致时拒绝并告警

### 敏感信息
- API Key、商户私钥存环境变量，不进代码仓库
- `raw_callback` 字段存储回调原文用于事后审计，但不对外暴露

## 优先级与依赖

- **优先级：P0** — 商业模式核心，影响平台收入入账
- **前置依赖**：需注册 Ping++ 账号并完成企业认证（约 3–5 个工作日，应提前启动）
- **后续影响**：
  - 文档 `06-数据看板.md` 的收入分析模块依赖 `payments` 表数据
  - 文档 `03-微信小程序.md` 的小程序支付需在此基础上增加 `wx_lite` channel 支持
- **风险**：商户资质审核周期不确定，建议并行推进营业执照准备
